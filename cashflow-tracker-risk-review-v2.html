CLOSED 1. Audit trail & transaction corrections

Problem

The original schema had no way to link a correction to the original transaction. Overwriting or deleting transactions would destroy the financial audit trail.

Solution

The system never hard-deletes or overwrites transactions. If a transaction is wrong, it is deactivated and a new correcting transaction is created to replace it. Every correction or deactivation must record the reason and the ID of the user who made the change. Admin view shows both active and inactive transactions, together with the full correction history.

Schema additions to transactions:

  • is_active BOOLEAN DEFAULT TRUE — soft-delete flag; all reports filter on WHERE is_active = TRUE
  • void_reason TEXT — mandatory when is_active is set to FALSE
  • voided_by INTEGER REFERENCES users(id) — who deactivated the row
  • replacement_transaction_id INTEGER REFERENCES transactions(id) — FK on the voided row pointing forward to the new correcting transaction, if one exists. Named from the perspective of the old row: "I was replaced by this transaction." The new row needs no back-reference — you can always find it via this FK.
Decision — closed Soft-delete with mandatory void_reason and user attribution. No hard deletes ever. Reports use active rows only. Admin sees full history.

CLOSED 2. Authentication & user identity

Problem

logged_by TEXT was free text — no enforcement, no permissions, no identity consistency between the web form and Telegram.

Solution

A small fixed users table holds records for the 3 known users. The web form uses a simple login page with username and password, one account per user. All transactions are linked to the authenticated user ID, never to a typed name.

For Telegram (Phase 5), access is restricted by storing each user's Telegram user ID directly in the users table as a nullable column (telegram_user_id TEXT UNIQUE). The bot checks incoming messages against this column — if the Telegram ID is not found, the message is ignored. Identity mapping is part of the user model, not the generic settings table.

ChannelAuth methodIdentity source
Web formUsername + password login pageusers.id from session
Telegram botTelegram user ID matched against users.telegram_user_idusers.id from matched row

The users table structure:

  • id INTEGER PRIMARY KEY
  • username TEXT NOT NULL UNIQUE
  • password_hash TEXT NOT NULL
  • telegram_user_id TEXT UNIQUE — nullable; populated when Telegram bot is set up in Phase 5
  • created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
Decision — closed Username + password login for web form. Telegram user ID stored directly in users.telegram_user_id — identity belongs to the user model, not settings. Both channels resolve to a single users.id. logged_by column stores users.id, not free text.

CLOSED 3. Database: sandbox vs production

Problem

The original plan said "SQLite → PostgreSQL later" with no clear boundary between test and production data.

Solution

PhaseDatabaseData statusPurpose
Sandbox (Phase 1)SQLiteNon-production — may be discardedValidate UX, logic, reporting
Go-livePostgreSQL on AzureProduction from this pointReal business data

Go-live is defined as the moment the system switches to PostgreSQL. The schema is identical between both engines — only the connection string changes. Migration is a one-time CSV export/import or pg_loader run taking roughly 1 hour. All data entered before go-live is considered non-production and may be discarded or migrated as needed.

Decision — closed SQLite is the sandbox only. Go-live = switch to Azure PostgreSQL. Data entered before go-live is non-production. From go-live onward, all data is real.

CLOSED 4. Category system

Problem

The concept doc listed example categories but had no fixed, versioned list. Free text or inconsistent values corrupt all GROUP BY reporting — "fuel" vs "petrol" vs "car fuel" aggregate separately and silently.

Solution

A categories reference table is defined before Phase 1 development starts. The transactions table stores a category_id integer FK, never a text string. The UI shows the display label.

CLOSED 5. VAT rate flexibility

Problem

Hardcoding vat_rate IN (0, 5, 8, 23) as a database CHECK constraint means any law change requires a schema migration.

Solution

The initial valid rates remain 0, 5, 8, and 23 for MVP. Validation moves to the application layer only, driven by a configurable reference list (settings table or vat_rates reference table). Adding a new rate becomes a data change, not a schema change.

Decision — closed Remove the CHECK constraint from schema. Validate VAT rates in application code against a configurable reference list. Initial values: 0, 5, 8, 23.

CLOSED 6. Income vs expense VAT logic — clean separation

Problem

Income and expense VAT logic was mixed into the same fields, creating ambiguity. They answer different business questions and must be kept separate.

Solution

Income transactions: use income_type (values: internal / external). Internal means no real sale — VAT is 0% and gross equals net. External defaults to 23% VAT, editable if needed. No VAT deductibility field on income rows.

Expense transactions: use vat_deductible_pct only. No internal/external concept on expenses. This field is mandatory and defaults to 100%.

Decision — closed Income uses income_type (internal/external). Expenses use vat_deductible_pct. These fields do not cross between directions.

Sources: Burak

CLOSED 7. Expense VAT deductibility — never null

Problem

A NULL vat_deductible_pct on an expense row silently breaks VAT reclaimable calculations — the formula returns NULL and the row disappears from aggregates without any error.

Solution

Expense transactions always require vat_deductible_pct. Default is 100%. User can change to 50% or 0%.

Enforcement is layered by phase:

  • SQLite sandbox (Phase 1): enforced in the UI and application layer only — SQLite does not support conditional CHECK constraints cleanly
  • PostgreSQL production (go-live): additionally enforced at the database level with a conditional CHECK constraint
-- PostgreSQL only — add at go-live
ALTER TABLE transactions
ADD CONSTRAINT chk_expense_vat_deductible
CHECK (direction != 'expense' OR vat_deductible_pct IS NOT NULL);
Decision — closed Expense transactions always have vat_deductible_pct. Default 100%. Allowed values: 100, 50, 0. Enforced in UI and app layer from day 1 (SQLite sandbox). DB-level conditional CHECK constraint added at go-live (PostgreSQL).

Sources: Claude, Burak

CLOSED 8. Gross amount always stored as-is

Problem

VAT deductibility should not affect the stored amount — otherwise the record no longer reflects the real cash movement.

Solution

The amount field always stores the actual gross cash amount paid or received. Deductibility affects derived reporting values only, never the stored amount.

ScenarioStored amountDerived: VAT reclaimableDerived: effective cost
Gross 123 PLN, VAT 23%, deductible 100%123.0023.00100.00
Gross 123 PLN, VAT 23%, deductible 50%123.0011.50111.50
Gross 123 PLN, VAT 23%, deductible 0%123.000.00123.00
Decision — closed Always store gross amount. Reclaimable VAT and effective cost are derived at query time, never stored.

Sources: Burak

CLOSED 9. Internal income rule — enforced, not suggested

Problem

If income_type = internal, the VAT rate must be 0% by definition. Without enforcement, a user could accidentally set a contradictory VAT rate.

Solution

When income_type is set to internal, the system automatically forces vat_rate = 0 and locks the field so the user cannot override it. Enforced in both the UI and backend validation.

Decision — closed income_type = internal forces vat_rate = 0. The field is greyed out and locked. This is a hard rule, not a default.

Sources: Burak

CLOSED 11. Mixed VAT invoices — advanced mode

Problem

Real Polish invoices occasionally contain items at different VAT rates (e.g. some lines at 23%, others at 8%). A single VAT rate field cannot correctly represent such an invoice, but full line-item accounting is out of scope for this tool.

Solution

Add an optional advanced VAT mode as an escape hatch. In normal mode, user selects a standard VAT rate. In advanced mode, user enters the total VAT amount manually. Gross amount remains the primary stored value. Derived calculations use the manual VAT amount when provided.

Normal mode (default)Advanced mode
VAT inputSelect rate: 0 / 5 / 8 / 23%Enter VAT amount manually
Gross amountAlways storedAlways stored
VAT calculationDerived from rate × grossUses manual VAT amount directly
Use caseAll standard transactionsMixed-rate invoices only
UI visibilityDefault — always shownHidden behind toggle — optional
Decision — closed Advanced VAT mode is optional and hidden by default. It exists only for mixed-rate or non-standard invoices. It must not complicate the normal workflow.

CLOSED 10. Input guardrails & validation

Problem

Users entering net amounts instead of gross, wrong VAT rates, or forgetting to log card commissions separately — these corrupt the data silently.

Required guardrails (Phase 1 — not optional)

  • Gross amount reminder — persistent label on the amount field: "Enter gross amount (VAT included)"
  • Category auto-defaults — selecting a category auto-fills the suggested VAT rate and, for expenses, the default deductibility percentage. User can override consciously
  • Card payment reminder — when payment_method = card: "Log gross amount. Card commission is logged separately at month end from terminal invoice"
  • Telegram confirmation step — always show extracted fields and ask for confirmation before saving. Non-negotiable, not optional

Unusual value detection (flag amounts outside historical range for the category) is a later-phase addition, not Phase 1.

Decision — closed Guardrails are Phase 1 requirements. Gross amount reminder and category-based defaults must be present from day 1.

Sources: ChatGPT, Burak

CLOSED 12. Opening balance — correction & validation

Problem

Opening balance was described as "sacred, log once, never reconstruct" but had no correction flow if someone set it wrong on day 1, and no validation that as_of_date precedes the first transaction date.

Solution

  • Opening balance is correctable by admin, but every correction must be logged with a reason and timestamp — same audit trail principle as transactions
  • Correction history stored in settings_audit table
  • System validates on startup that as_of_date <= MIN(transaction date) and warns if not
  • Cash flow statement is blocked if no opening balance is set
Decision — closed Opening balance is mandatory, audited, and correctable with traceability. Cash flow report is blocked until it is set.

CLOSED 14. Phase ordering & MVP assumption

Situation

Phase 1 (web form) comes weeks before Phase 5 (Telegram bot). Two of the three users may prefer Telegram, but they have no option during early phases.

Decision — closed Web form is the sole input method during Phases 1–4. All 3 users use the web form. Telegram arrives in Phase 5. This is a conscious, accepted limitation — not an unstated assumption.

CLOSED 15. Transaction ordering & "fix last" logic

Problem

Multiple transactions can be entered on the same calendar date. Without precise ordering, "fix last" and "delete that" Telegram commands become ambiguous.

Solution

Use created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP as the authoritative ordering field — already in the schema. All correction logic and "last transaction" queries must order by created_at DESC, not by date DESC.

Decision — closed created_at timestamp is the real ordering mechanism. Date alone is not sufficient. Correction and audit queries always use ORDER BY created_at DESC.

Sources: Burak

CLOSED 16. Final VAT model — consolidated specification

This is the authoritative VAT specification. All agents and builders must follow this. It supersedes any earlier descriptions in concept.md or architecture.md.

All transactions

  • amount is always stored as gross (the actual cash amount)
  • vat_rate is always stored in normal mode
  • manual_vat_amount can optionally be stored in advanced mode for mixed-rate invoices

Income transactions

  • Use income_type: internal or external
  • internal → VAT forced to 0%, gross = net, field locked
  • external → VAT defaults to 23%, user can change
  • No vat_deductible_pct field on income rows

Expense transactions

  • No income_type field on expense rows
  • vat_deductible_pct is mandatory — default 100%, allowed values: 100 / 50 / 0
  • Petrol / car: 50% by default (Polish rule)
  • Meals / entertainment: 0% by default (not deductible in Poland)

Derived calculations (never stored)

vat_amount       = amount - (amount / (1 + vat_rate / 100))
               OR = manual_vat_amount  (if advanced mode used)

net_amount       = amount - vat_amount
vat_reclaimable  = vat_amount × vat_deductible_pct / 100   (expenses only)
effective_cost   = net_amount + (vat_amount × (1 - vat_deductible_pct / 100))
net_vat_position = SUM(income vat_amount) - SUM(expense vat_reclaimable)
Decision — closed This is the final VAT structure. Income logic and expense logic are separate. Derived values are never stored. Advanced mode is an optional escape hatch for edge cases only.

Sources: Burak

CLOSED 17. Practical simplicity rule

Principle

The main strength of this project is simple daily usage. Advanced features must never complicate the normal workflow for everyday users.

Standard transaction flow (must remain this fast)

  1. Choose category
  2. Choose income or expense
  3. Enter gross amount
  4. Accept suggested VAT defaults
  5. Adjust only when necessary

Advanced VAT mode, unusual value warnings, and admin views are all hidden unless needed. They support edge cases without surfacing into the daily flow.

Decision — closed Normal workflow stays fast and simple. Advanced capabilities are available but invisible by default.
