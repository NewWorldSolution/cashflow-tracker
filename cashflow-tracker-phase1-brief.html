02

Phase 1 Scope

Core problem

A small Polish service business has no reliable way to log daily income and expenses with correct VAT logic, card reconciliation awareness, and a traceable audit trail. Notebook and spreadsheet both fail on VAT correctness.

Target user

3 people — owner, assistant, wife. Non-technical. Need fast daily entry. Must feel simple on first use. Polish VAT rules apply. All 3 use the web form in Phases 1–4. Telegram arrives in Phase 5.

Phase 1 definition

All 3 users can log any transaction via web form. Data stored correctly: gross, VAT, category FK. Corrections possible with full audit trail. Opening balance set. Running on SQLite locally.

Phase 1 is the sandbox. SQLite only. All data is non-production and may be discarded at go-live. The app must show a visible SANDBOX banner on every page so no one treats this data as real. Go-live = switch to Azure PostgreSQL — schema is identical, only the connection string changes.

04

Technical Direction

Stack — locked per CLAUDE.md
  • Python + FastAPI — async, clean REST, easy Azure deploy, Pydantic validation built-in
  • SQLite — sandbox only; schema identical to PostgreSQL production target
  • Jinja2 — server-side templates; no SPA; no client-side framework
  • Vanilla JS — form behaviour only: auto-defaults, field locking, banners
  • bcrypt — password hashing; cookie-based sessions via FastAPI middleware
No architecture changes needed

The architecture is fully decided. Schema, stack, VAT model, category list, and auth are all closed decisions. The task for Phase 1 is implementation — not design.

The one open question in CLAUDE.md is Flask vs FastAPI. FastAPI is recommended: async by default, Pydantic validation reduces boilerplate, cleaner foundation for Phase 5 API expansion to Telegram.

Critical rule — must not be broken

Every business rule must be enforced in the FastAPI route handler — not only in the HTML form. The form is a convenience layer. The API is the contract. Rules enforced only in the UI will be silently bypassed when Phase 5 (Telegram) and Phase 6 (LLM) skip the form entirely.

Rules that must live in the API: income_type=internal → vat_rate=0 · expense → vat_deductible_pct NOT NULL · other_* → description NOT NULL · amount > 0 · vat_rate IN (0,5,8,23) · void → void_reason NOT NULL + voided_by

Technical risks
SQLite constraint gap
SQLite does not support conditional CHECK constraints. The vat_deductible_pct NOT NULL on expenses constraint must be enforced in the app layer only during sandbox. The PostgreSQL CHECK is added at go-live — flagged in architecture.md with a comment.
Session persistence
Cookie-based sessions need a stable secret key in .env. If the key is not persisted, every redeploy invalidates all sessions. Configure this correctly from day 1.
Seed idempotency
If seed runs twice, duplicate category rows corrupt the FK reference table. Use INSERT OR IGNORE or existence checks in the seed script.
Opening balance gate
A reminder banner can be dismissed. The form must be hard-blocked — redirect to setup screen — until opening_balance exists in the settings table. A warning is not sufficient.

Success Criteria

All 3 users can log in with separate credentials and maintain independent sessions
A transaction can be entered for every combination of direction × category × payment method without a form or API error
Selecting internal_transfer income category automatically sets VAT to 0% and the field is locked — user cannot override it in the UI or via a direct API call
Submitting an expense transaction without vat_deductible_pct is rejected by the API with a descriptive error — it is never silently saved with NULL
Submitting other_expense without a description is rejected at the API level — a direct POST bypassing the form also fails
Card payment method shows the commission reminder banner
Voiding a transaction without a reason is rejected — the row is never deactivated without void_reason and voided_by
Voided transaction disappears from the default list — it is not hard-deleted and remains in the database with is_active = FALSE
Opening balance setup screen appears on first visit and the main form is hard-blocked until it is set
Every stored transaction has logged_by as a valid integer FK to users.id — no text strings in that column, verified by direct DB query
Every stored transaction amount is the gross cash amount — no net amounts in the database
SANDBOX banner is visible on every page while SQLite is active
07

Risks & Failure Modes

Business logic in UI only
Validation only in the HTML form works in Phase 1 but is silently bypassed by Phase 5 Telegram and Phase 6 LLM extraction. This is the highest-probability failure mode. Every rule must live in the FastAPI handler, not the template.
Opening balance skipped
Users dismiss banners. The form must hard-redirect to the setup screen — not show a warning. Without opening balance, Phase 7 cash flow statement is permanently incorrect and cannot be fixed retroactively.
Category as free text
Any code path that accepts a category name string instead of category_id integer destroys reporting consistency permanently. Schema FK is the backstop — but the API must also reject category strings explicitly.
Real data in sandbox
If any user starts entering real transactions during Phase 1 SQLite testing, go-live migration becomes complicated and trust in the data becomes unclear. SANDBOX banner is not decoration — it sets user expectations.
Over-engineering the UI
This is a 3-user internal tool on a local SQLite sandbox. Time spent on visual polish, responsive layouts, or JS enhancements is wasted Phase 1 time. The form must be correct and fast to use — not beautiful.
Hard deletes in sandbox "just this once"
Bad patterns introduced in sandbox survive into production. The soft-delete model must be implemented correctly from the first transaction. Any code that calls DELETE on a transaction row is wrong and must be caught in code review.
Agentic session drift
Claude Code sessions that run without re-reading CLAUDE.md will drift from the rules — adding free-text fields "for now", skipping void_reason "it's just sandbox", or proposing a JS framework. Every session must start with project.md and CLAUDE.md.
08

Brutal Feedback

This project has genuine real-world value — precisely because it refuses to be generic.

Most small Polish service businesses run their finances on a notebook, a spreadsheet, or nothing at all. Their accountant gets a bag of receipts at month end. The owner has no idea what their VAT position is mid-month, card reconciliation is a mystery, and every correction is done by crossing out a line. This tool solves that specific, painful, real problem — for one business, with domain precision.

The VAT model is the strongest part of the entire design. Always gross, derived calculations never stored, income_type separated from expense deductibility, the internal income rule as a hard enforced constraint — this is the kind of precision that generic tools get wrong. Fakturownia, inFakt, and every Polish SMB accounting SaaS either requires an accountant to configure, conflates VAT tracking with tax submission, or silently stores net amounts. This tool does none of that, and it knows what it is.

The real differentiator is not Phase 1. The web form is a prerequisite. The actual product is Phase 5 + 6: a Telegram group bot where the owner sends a voice note or a receipt photo and it is correctly extracted, VAT-assigned, confirmed, and saved in 15 seconds. That is the thing no existing Polish SMB tool does at all, and the data model built here is exactly right to support it.

What would waste the most time: polishing the Phase 1 UI, debating Flask vs FastAPI for more than one hour, adding features not in the brief, or allowing any session to proceed without re-reading CLAUDE.md. The design is genuinely good. The failure mode is implementation discipline, not design quality.

What would make it competitive beyond one business: the skill layer and CLAUDE.md approach is the real transferable asset here. This is a replicable framework for building private, domain-correct financial tools agentically — the cashflow tracker is the first instance. If it works cleanly, the same pattern applies to any Polish SMB with different category lists and VAT rules. That is the product if there ever is one.

Bottom line: Build Phase 1 exactly as specified. Do not add features. Do not polish. Get all 3 users entering real transactions for 2–4 weeks on the sandbox, then go-live on PostgreSQL. Phase 5 is where this becomes genuinely useful every day. Phase 6 is where it becomes hard to replicate.
